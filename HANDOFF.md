## What happened this session (2026-07-20, v15)

M5 finished last session (v14); this session started M6, the next milestone in `ARCHITECTURE.md`. One commit (`termite-terminal`/`termite-app`, see git log — HANDOFF is written before the commit lands, check `git show --stat` for the exact hash).

**First finding: two M6 checklist items were already done.** Before writing any new code, a read of `crates/termite-terminal/src/{grid,handler}.rs` turned up that alternate-screen-buffer (`?1049h`/`?1049l`) and `xterm`-style title setting (OSC `0`/`2`) were both implemented back in M1 (the handler even has a stale doc comment: "Only `?1049h`/`?1049l` matter for M1"). Neither was ever marked done in the milestone table, since M6 hadn't been reached yet. Left in place, comment updated to describe the current (post-this-session) DEC-private-mode dispatch instead of the M1-era one.

**Real window-size-driven resizing**: `TermiteApp` gained a `grid_size: (usize, usize)` field, updated by a new `Message::WindowResized(iced::Size)` fed from `iced::window::resize_events()` (added to `subscription()`). `grid_size_for_window` (pure function, unit-tested directly) converts pixels to a row/column count using approximate cell metrics (`CELL_WIDTH_PX`/`CELL_HEIGHT_PX`, tuned for `Font::MONOSPACE` at the terminal's text size 14) and fixed chrome sizes (`SIDEBAR_WIDTH_PX` matching `sidebar::view`'s literal `Length::Fixed(220.0)`, `CHROME_HEIGHT_PX` an estimate for the tab bar) — **not** measured against the real font, since Iced has no synchronous glyph-metrics query available at the app layer; a deliberate approximation, flagged in the doc comment. `resize_all_tabs` then resizes every open tab's grid and propagates to whatever's feeding it: `Pty::resize` for a local tab (already existed, just never called), `SessionCommand::Resize` for a live SSH tab (also already existed end-to-end in `termite-ssh` — `session.rs` was already handling the command, `termite-app` just never sent it). `Tab::new`/`spawn_local_tab` both gained `rows`/`cols` parameters (previously hardcoded to the module-level `ROWS`/`COLS` constants) so a tab opened after a resize is created at the current size rather than the M1-era default. `ROWS`/`COLS` still exist as the fallback size before the first `WindowResized` event arrives (Iced has no synchronous "current window size" query at startup either).

**Bell**: `TerminalGrid` gained a `bell: bool` flag set by `ring_bell()` (wired to `execute(0x07)` in `handler.rs`) and cleared by `take_bell()`. `Tab::advance` now returns whether a bell rang during that batch of bytes; both callers that feed real shell/session output (`Message::PollOutput`'s loop, `SessionEvent::Output` in `handle_session_event`) check it and, if true, set a new `Tab::bell_flash` field and schedule `Message::BellTimeout(tab_id)` after `BELL_FLASH_DURATION` (200ms) via a `bell_timeout_task` helper — same `Task::perform(async move { sleep(...).await }, ...)` shape as M5's reconnect backoff, for the same reason (constructing `sleep` outside the async block panics outside a Tokio runtime, which is exactly what a plain `#[test]` is). `view()` wraps the terminal pane in a `container` with a 2px `ACCENT`-colored border when the active tab's `bell_flash` is set, cleared on `BellTimeout`. Deliberately *not* on `TerminalGrid`: whether the flash is currently showing is UI-timing state, not something the VT-parsing layer should own.

**Bracketed paste mode**: `TerminalGrid` gained `bracketed_paste: bool`, toggled by CSI `?2004h`/`?2004l` (extended `private_mode_dispatch` in `handler.rs`, which previously only handled `?1049`). Paste itself is `Ctrl+Shift+V` (checked in `is_paste_shortcut`, intercepted in `KeyPressed` before the shortcut/byte-forwarding path) — deliberately not plain `Ctrl+V`, which `key_to_bytes` already forwards as control byte `0x16` (readline's "quoted insert"; claiming it would be a real regression, unlike the Ctrl+Tab/Ctrl+digit precedent from M5). The shortcut returns `iced::clipboard::read().map(Message::Pasted)`; `Message::Pasted(Some(text))` routes to `paste_into_active_tab`, which wraps `text` in `ESC[200~`/`ESC[201~` iff the active tab's grid has bracketed paste enabled, then sends it exactly like `KeyPressed` does (`writer.write_all` for a local tab, `SessionCommand::Write` via `send_to_session` for SSH).

**Mouse reporting**: `TerminalGrid` gained a `MouseTracking` enum (`Off`/`X10`/`Normal`/`ButtonEvent`/`AnyEvent`) plus a `mouse_sgr: bool`, driven by `?9`/`?1000`/`?1002`/`?1003` (tracking mode — mutually exclusive, enabling one clears any other rather than layering) and `?1006` (SGR coordinate encoding, independent of tracking mode) in `private_mode_dispatch`. Capture on the UI side uses Iced's `mouse_area` widget wrapped around the terminal pane's `container`, whose `on_press`/`on_release`/`on_move`/`on_scroll` callbacks don't carry a position — `TermiteApp` now tracks `mouse_position`/`mouse_button_down` itself (updated by a new `on_move`) so a press/release/scroll can look up where the cursor already was. `report_mouse_event` gates everything on the active tab's `mouse_tracking` (nothing sent at all when `Off`; a plain, buttonless motion report additionally requires `AnyEvent`, a drag motion report requires `ButtonEvent` or `AnyEvent`) then calls `encode_mouse_event`, which emits either the SGR form (`ESC[<Cb;Cx;CyM`/`...m`, unbounded coordinates, distinguishes which button was released) or the legacy 6-byte form (`ESC[M` + three bytes each offset by 32, released button unrecoverable — always reported as the fixed "no button" code, and coordinates capped at 223 since the offset byte can't exceed 255). Pixel-to-cell conversion (`cell_at`) reuses the same approximate `CELL_WIDTH_PX`/`CELL_HEIGHT_PX` constants as `grid_size_for_window`, clamped to the grid's current bounds so a cursor sitting past the visible edge doesn't report an out-of-range cell. Deliberately unsupported: modifier-key bits in the button code (Iced's `mouse_area` callbacks don't surface `Modifiers`, and threading that through wasn't judged worth it yet for a first pass) and X10 mode's release-suppression nuance (X10 technically never reports release at all; this implementation reports it the same as `Normal` — a state a real X10-only client would just never trigger, since it doesn't expect a release byte, but worth knowing if it turns out to matter).

**Tests**: 19 new in `termite-app` (61 → 80): resize/bell/paste tests from the first half of the session (see prior paragraphs), plus `cell_at`'s pixel math and edge-clamping, `encode_mouse_event`'s SGR press/release distinction, legacy 32-offset encoding and 223 coordinate cap, and `report_mouse_event`'s gating behavior end-to-end through `update()` — tracking off is a no-op, `Normal` tracking reports press/release, plain motion only reports under `AnyEvent`, drag motion reports under `ButtonEvent`, wheel direction follows scroll sign. 6 new in `termite-terminal` (0 → 6): BEL/`?2004`/`?1049` from the first half, plus the four tracking-mode CSI sequences dispatching to the right `MouseTracking` variant, disabling any one mode resetting tracking to `Off`, and `?1006` toggling SGR encoding independent of tracking mode.

**Rendering verification** (Xvfb `:99`, same method as v13/v14): confirmed the app still renders correctly, twice — once after wrapping the terminal pane in a `container` (bell), once more after adding the `mouse_area` wrapper (sidebar, tab bar, shell prompt all present both times, no visual change from either wrapper in their default/no-bell state, as expected since neither renders anything by itself). Same standing limitation as always: no way to simulate an actual mouse click/drag/scroll or keystroke through the isolated `:99` display to visually confirm a resize, flash, paste, or mouse report end-to-end — see "Isolated GUI testing" below for why. The `report_mouse_event`/`encode_mouse_event` tests above are the substitute, same as click-driven tab-bar interaction has relied on unit tests since M5.

**Not done / explicitly out of scope this session**: OSC 8 hyperlinks, selection modes (character/word/line), scrollback search, split terminal — the remaining 4 of M6's 9 checklist items. OSC 8 in particular was scoped out deliberately: `Cell` is `Copy` today (cheap to clone into `Vec<Vec<Cell>>` scrollback), and a hyperlink URL is unbounded-length data that doesn't fit that shape without either boxing every cell or a separate per-line range index — a real design decision, not a quick add, deferred rather than rushed. Selection modes are the natural next step now that mouse events are captured (`report_mouse_event`'s press/release/motion plumbing is most of what selection needs too, modulo actually tracking a selection range instead of encoding a report), but weren't started this session. Split terminal still needs UI-layer pane-layout work beyond what a single-`Text`-widget terminal pane currently has.

## Project history (v5–v14, condensed)

- **M0–M2** (earliest sessions): workspace scaffold + CI, local terminal emulator (PTY/VT/Iced), SSH core with mandatory `known_hosts` verification. All verified end-to-end; not revisited since.
- **M3 protocol layer** (v5–v6): RSA hash-algorithm threading (`rsa-sha2-256`/`512`, negotiated not guessed), SSH agent auth (`$SSH_AUTH_SOCK`, key material never enters the process), `~/.ssh/config` parsing (own parser, no crate dep — OpenSSH resolution semantics: first-value-wins, `IdentityFile` accumulates, fnmatch patterns, `Match` blocks inert). **Load-bearing gotcha**: `KeyProviderSigner::auth_sign` (`crates/termite-ssh/src/signer.rs`) must return the *entire* `to_sign` buffer with the signature appended as a length-prefixed SSH string, not bare signature bytes — russh slices the userauth packet out of it; getting this wrong hangs the session silently.
- **CI** (v6–v7): `rustsec/audit-check@v2` has its own `ignore:` action input separate from `.cargo/audit.toml` — the two must be kept in sync manually (now also documented in `CLAUDE.md`). The action also needs an explicit `checks: write` permission or it 403s publishing results even when the audit itself passes. Lesson: a commit message claiming a GitHub Actions change is "verified" isn't evidence — check the actual run (`gh` isn't installed here; unauthenticated `curl` against `api.github.com` works for public-repo run/job status).
- **M4 UI, part 1** (v7–v9): `HostStore` (TOML-backed, re-reads the whole file every call rather than caching, to avoid staleness vs. hand-edits), the sidebar list/add/delete, `SessionCommand`/`SessionEvent` wiring via a persistent Iced subscription (`ssh_worker`), then the auth-method picker + credential/host-key prompt modal (`AuthRequired`/`HostKeyUnknown`/`HostKeyMismatch` previously failed closed with no UI at all), then a save-to-keychain toggle on that prompt.
- **M4 UI, part 2** (v10–v12): `~/.ssh/config` `Host`-alias resolution in the add-host form (Enter on the address field), then deleting a host also forgetting its saved keychain credential (this closed a real gap: `CredentialStore` had no `delete_passphrase` at all before v12, only `delete_password`).
- **M4 completion** (v13): `HostProfile` gained `favourite: bool`/`last_connected: Option<u64>`; the sidebar got favourites, recency-ordered sorting (`sort_hosts()`), search/filter, host tags in the form, a profile editor (`EditHost`/`SaveHost` create-or-update), `~/.ssh/config` bulk import (`SshConfig::host_aliases()`), and key-generation UI (`generate_and_save_key`, never overwrites an existing file). Rendering verification caught a real bug here: the ★/☆ favourite glyphs don't exist in iced's default UI font and drew as missing-glyph boxes — replaced with plain ASCII `"*"`/`"o"`. This is why M5's tab-status glyphs went straight to ASCII instead of unicode.
- **M5** (v14): `TermiteApp` went from a single `grid`/`parser`/`writer`/`_pty`/`active_session: Option<SessionId>` to `tabs: Vec<Tab>` + `active_tab: Option<TabId>`, each `Tab` owning its own `TerminalGrid`/`vte::Parser` and a `TabKind` (`Local`/`Ssh`). New `TabId` type (mirrors `SessionId`'s shape, never persisted) identifies a tab independent of whichever `SessionId` it's currently bound to, since a reconnect gets a new `SessionId` but keeps the same `TabId`. Added: a tab bar (`crates/termite-ui/src/tabbar.rs`) mirroring `sidebar.rs`'s contract, per-tab event routing (`find_tab_by_session_mut`) replacing the old "active session wins" gate that silently dropped background output, `Ctrl+Tab`/`Ctrl+1-9` keyboard navigation, and reconnect with exponential backoff (2s/4s/8s/16s/30s capped) plus a manual retry button once `MAX_RECONNECT_ATTEMPTS` (6) is exhausted. **Load-bearing gotcha**: `tokio::time::sleep(delay)` must be constructed *inside* the `async move` block passed to `Task::perform`, not built eagerly and passed in directly — it registers with the Tokio timer driver the instant it's called, which panics outside a runtime context (exactly what a plain `#[test]` is). Not done: tab drag-to-reorder (no drag infrastructure exists anywhere in the codebase).
- **Testing methodology, established v7–v8 and unchanged since**: no live SSH server here (`sshd` exists but its systemd service is deliberately left untouched — shared system state); `termite-ssh`'s hermetic integration tests substitute. No click-driven GUI testing — see "Isolated GUI testing" below. Handler functions (`handle_session_event`/`update_sidebar`/`update_prompt`/`update_tabbar` as of M5) are instead exercised directly against a real `TermiteApp` with a fake `ssh_worker` channel standing in for the subscription.

**Isolated GUI testing** (unchanged since v7, still the only way to visually verify anything): running the app for a visual check must never put a window on the user's real desktop — it did once, by accident, early in this project, and was killed immediately once noticed. The safe pattern: `Xvfb :99 -screen 0 1280x800x24` (via the Bash tool's `run_in_background`, not shell `&`/`disown` — raw `&` backgrounding in this sandbox produces a spurious exit 144 on the whole tool call), then `WAYLAND_DISPLAY= DISPLAY=:99 WINIT_UNIX_BACKEND=x11 ./target/debug/termite`, screenshotted via `DISPLAY=:99 import -window root`. To seed host-list state for a screenshot without touching the user's real config, launch with `XDG_CONFIG_HOME=<scratch dir>` pointing at a scratch directory containing a hand-written `termite/hosts.toml` (new this session — see above). **Never use `ydotool`** to simulate clicks — it injects through `/dev/uinput`, a system-wide kernel input device not scoped to the `:99` display, so it would leak input into the user's real session. Only rendering can be verified this way, not click-driven interaction; no substitute has been found.

## What this project is

**Termite** — a modern, open-source, native SSH client built entirely in Rust.
- No accounts, no telemetry, no subscriptions, no AI, no cloud.
- Target aesthetic: VS Code / Warp / Obsidian — not PuTTY.
- Cross-platform: Linux, macOS, Windows.
- Philosophy: "It should disappear and let you work."

Full design rationale is in `ARCHITECTURE.md`. Read that first. Also read `CLAUDE.md` — command reference and security invariants, kept current.

## Suggested next steps

1. **Continue M6** — 5 of 9 checklist items done. Remaining: OSC 8 hyperlink support (needs a design decision first — `Cell` is `Copy` today and a URL doesn't fit that shape cheaply; see "Not done" above), selection modes (character/word/line — now that mouse capture exists via `report_mouse_event`'s press/release/motion plumbing, this is mostly "track a range and highlight it" rather than needing new input infrastructure), scrollback search (regex), split terminal (horizontal/vertical — needs pane layout the UI doesn't have yet, currently one `Text` widget per tab). Selection modes are probably the natural next pick, since the mouse-event plumbing they need just landed.
2. **Mouse reporting doesn't encode modifier keys** — Iced's `mouse_area` callbacks (`on_press`/`on_release`/`on_scroll`) don't surface `Modifiers`, so `report_mouse_event`'s button codes never set the shift/meta/ctrl bits real xterm mouse reports support. Would need either a lower-level event subscription than `mouse_area` gives, or tracking modifier state from the existing `KeyPressed` subscription and reading it back in on press — not attempted this session, flagged as a known gap.
3. **`grid_size_for_window`/`cell_at`'s cell-metric constants are an approximation, not measured** — `CELL_WIDTH_PX`/`CELL_HEIGHT_PX` in `termite-app/src/lib.rs` are eyeballed for `Font::MONOSPACE` at size 14, not queried from Iced (no synchronous glyph-metrics API available at this layer). Now feeds both window resizing *and* mouse-to-cell conversion, so a mismatch here would misdirect both. Worth revisiting if the terminal pane visibly over/under-fills the window, or clicks land on the wrong cell — hasn't been checked against a real font render or real click, only unit-tested against its own math.
4. **Tab drag-to-reorder** — still the one M5 checklist line not done; there's no drag infrastructure anywhere in this codebase yet. Low priority unless it turns out to matter more in practice than it seems right now (tabs can still be opened/closed/switched freely, just not reordered).
5. **End-to-end verification against a real server** — still nothing has exercised `AuthRequired`/`HostKeyUnknown`/`HostKeyMismatch`, the tab/reconnect machinery, or now the resize-over-SSH path (`SessionCommand::Resize`), against an actual `sshd`. A user-scoped `sshd -f <config> -p <high port> -D` bound to `127.0.0.1` with its own host key and `AuthorizedKeysFile` would be hermetic and worth trying — same spirit as `termite-ssh`'s existing in-process test agent/server.
